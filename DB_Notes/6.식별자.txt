1. 식별자(Identifier) 개념
    1) 엔터티는 인스턴스들의 집합임
        집합내에서 단 하나의 인스턴스를 구별해 낼 수 있는 논리적인 이름이 필요하며 이러한 구분자를 식별자라고 함
    2) 하나의 속성 혹은 속성들의 조합을 식별자로 정의함
    3) 엔터티 내에서 하나의 행을 콕 집어낼 때 사용하는 것임

2. 식별자 예시
    1) 고객(고객번호)
    2) 사원(사원번호)
    3) 주문(주문번호)
    4) 상품(상품번호)


3. 식별자의 분류
    1) 주 식별자(대표성)
        -엔터티 내에서 각각의 행을 구분할 수 있는 구분자임
        -다른 엔터티와 참조관계를 가질때 연결할 수 있는 식별자임
        -예) 사원번호, 고객번호
       보조 식별자
        -주식별자가 아니라서 대표성을 가지지 못함
        -다른 엔터티와 참조관계를 가질때 연결할 수 없음
        -예) 주민등록번호

    2) 내부 식별자 (스스로)
        -엔터티 내부에서 스스로 만들어지는 식별자
        -예) 고객번호
       외부식별자
        -다른 엔터티와의 관계를 통해 다른 엔터티로부터 받아오는 식별자
        -예) 주문엔터티의 고객번호

    3) 단일 식별자(속성(#)의 숫자)
        -하나의 속성으로 구성된 식별자
        -예) 고객 엔터티의 고객번호
       복합 식별자
        -둘 이상의 속성으로 구성된 식별자
        -예) 주문상세 엔터티의 주문번호+상세순번

    4) 본질 식별자
        -비즈니스에 의해 만들어지는 식별자
        -예) 고객번호
       인조 식별자
        -본질 식별자가 복잡한 구성을 있기 때문애 인위적으로 만든 식별자
        -예) 고객번호+주문순번+...(복잡) => 주문엔터티의 주문번호

        엔터티      속성                   식별자 분류
        ---------------------------------------------------------------------
        부서        부서번호           주식별자,내부식별자,단일식별자,인조식별자
        사원        사원번호           주식별자,내부식별자,단일식별자,본질식별자
        사원        주민등록번호       보조식별자,
        사원        부서번호           외부식별자
        교육이력    사원번호+수강일자   주식별자,복합식별자,본질식별자
        구매신청    구매신청번호        주식별자,단일식별자,인조식별자


4. 식별자 특징
    1) 엔터티 내에 존재하는 각각의 인스턴스 집합은 주식별자에 의해 유일하게 구분할 수 있음(유일성)
        -예) 사원 엔터티의 사원번호 속성은 주식별자임
              사원번호는 모든 지원들에 대해 개인별로 고유하게 부여함

    2) 유일성을 만족한다면 주식별자를 구성하는 속성의 수는 최소한의 수로 이루어져야 함 (최소성)
        -예) 사원분류코드 + 사원번호 <== 부적절한 주식별자 구조임

    3) 엔터티 내 특정 인스턴스에 주식별자가 한 번 정해지면 
        그 값은 변하지 말아야 한다 (불변성)

    4) 주식별자가 지정되면 반드시 데이터 값이 존재해야 함 
        (NULL 허용불가) (존재성)
        -NULL : 아직 정해지지 않은 미지의 값



5. 식별자 도출 기준
    1) 비즈니스에서 자주 이용되는 속성을 주식별자로 지정함
    2) 명칭, 장소과 같이 이름으로 기술 되는 속성은 가능하면 주식별자로 하지 않음
    3) 주식별자를 복합식별자로 할 경우 지나치게 많은 속성이 포함되지 않도록 함



6. 식별자 관계 & 비식별자 관계
    1) 식별자 관계
        -자식 엔터티에서 부모 엔터티로부터 받은 외부 식별자를 
          (자식)자신의 주식별자로 이용한 경우.

    2) 비식별자 관계
        -자식 엔터티에서 부모 엔터티로부터 받은 외부 식별자를
          부모와 연결이 되는 속성으로서만 이용할 경우

    3) 식별자 관계를 사용할지 비식별자 관계를 사용할지는 업무특징, 자식 엔터티의 주식별자 구성, 작성 전략에 의해 결정함

    4) 식별자 관계로만 각각의 엔터티 간의 관계를 정의한 모델
        -관계가 도출될때마다 PK 속성의 수가 지속적으로 증가
        -SQL문의 복잡도가 올라가며 간혹 조인 조건을 누락하는 실수 발생가능

    5) 데이터 모델링 작업 시 식별자 관계와 비식별자 관계를 취사선택하여 연결하는 것
        -높은 수준의 내공을 필요로 하는 데이터 모델링 기술임
        -기본적으로 식별자 관계로 모든 관계를 연결하면서 아래 조건에 해당하는 경우 비식별자 관계로 조정하면 됨
        -관계 강/약 분석 => 약한(긴밀X)관계 => 비식별자 관계 고려
        -자식테이블에 독립 PK 필요 => 독립된 PK구성 => 비식별관계 고려
        -SQL복잡성 증가(개발 생산성 하락) => PK속성 단순화 => 비식별자 고려
